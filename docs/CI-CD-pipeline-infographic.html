Lint, test, scan, build, publish

Runs on every push and every pull request. Nothing gets built unless it passes first.

  • Ruff — ruff check backend
  • Pytest + coverage, gated at 80% — coverage report --fail-under=80
  • Bandit security scan — bandit -r backend --exclude backend/tests
  • Build 3 images: ros2-monitoring, backend, frontend
  • Push to DockerHub, tagged with the commit SHA (push events only)

3
github repository · branch: master

Git becomes the desired state

helm/ros2-cicd/values.yaml on master is, at any moment, the full definition of what should be running. Not a deploy log — the actual source ArgoCD reads from.

why pull, not push

GitHub's cloud runners have no route to a local minikube cluster. Rather than hand CI a kubeconfig, the cd job only ever writes to git — ArgoCD, running inside the cluster, is the one that reaches out and pulls. Cluster credentials never leave the cluster.

4
argocd · application: ros2-cicd

Reconcile the cluster to match git

Polls the repo, renders helm/ros2-cicd itself (no helm binary needed in-cluster), and applies through the same Kubernetes API kubectl uses.

  • Sync policy: automated, prune on
  • selfHeal on — a manual kubectl edit gets reverted back to match git
  • Destination namespace: cicd-demo
5
kubernetes · namespace: cicd-demo

Traffic reaches only what's actually ready

New pods roll in behind the old ones and only take traffic once they pass their own readiness check — not just "process started."

  • frontend × 2 replicas
  • backend × 2 replicas — readiness gated on actually discovering the ros2 pod over DDS
  • ros2 × 1 replica — publisher_node + subscriber_node, sharing ROS_DOMAIN_ID with the backend
